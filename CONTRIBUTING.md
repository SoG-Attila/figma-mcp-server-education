## ü§ù Ways to Contribute

### 1. Share Your Experience

**Teaching Stories:**
- What worked well in your classroom?
- What challenges did you face?
- How did you adapt the materials?
- What timing adjustments did you make?

**Where to share:** Open a Discussion on GitHub

### 2. Report Issues

**Found a problem?**
- Typo or error in documentation
- Technical step that doesn't work
- Confusing instruction
- Missing prerequisite
- Outdated screenshot or example

**How to report:** Open an Issue on GitHub with:
- Clear description of the problem
- Which file/section it's in
- Your OS and version (if technical)
- Screenshot if helpful

### 3. Suggest Improvements

**Ideas for enhancement:**
- Additional example prompts
- New use cases
- Better explanations
- Alternative teaching approaches
- Extension projects
- Translation to another language

**How to suggest:** Open an Issue labeled "enhancement"

### 4. Submit Code or Documentation

**You can improve:**
- Tutorial clarity
- Server code efficiency
- Error messages
- Configuration examples
- Troubleshooting guides
- Example files

**How to submit:** Fork, make changes, submit Pull Request

---

## üìù Contribution Guidelines

### Documentation Changes

**Style Guide:**
- Clear, concise language
- Assume no technical background
- Use analogies and examples
- Break complex steps into smaller ones
- Include success indicators
- Add troubleshooting tips

**Format:**
- Markdown for docs
- HTML for tutorials (with inline CSS)
- Comments in code
- Screenshots when helpful

### Code Changes

**Server Code:**
- Keep it simple and readable
- Add comments for learning
- Maintain compatibility with tutorial
- Test on Windows and Mac
- Don't add unnecessary complexity

**Configuration:**
- Clear examples
- Comment every field
- Show both Windows and Mac paths
- Include validation tips

### New Examples

**Prompt Examples:**
- Test with actual Figma file
- Show expected output
- Explain what makes it effective
- Cover different use cases
- Note any limitations

**Figma Templates:**
- Universal business applications
- Clear structure
- Placeholder text that guides
- Both simple and complex versions

## üîÑ Pull Request Process

### Before Submitting

1. **Test your changes:**
   - If code: Test on Windows and Mac
   - If docs: Have someone else read it
   - If tutorial: Walk through every step

2. **Check compatibility:**
   - Works with existing materials
   - Doesn't break current functionality
   - Maintains beginner-friendly approach

3. **Document your changes:**
   - What problem does this solve?
   - How is it better?
   - Any new dependencies?

### Submitting

1. **Fork the repository**
2. **Create a feature branch**
   ```bash
   git checkout -b feature/your-improvement
   ```
3. **Make your changes**
4. **Commit with clear messages**
   ```bash
   git commit -m "Improve troubleshooting section for Windows"
   ```
5. **Push to your fork**
6. **Open Pull Request**

### PR Description Template

```markdown
## Description
Brief explanation of what you changed and why.

## Type of Change
- [ ] Bug fix (tutorial error, broken link)
- [ ] Enhancement (improved explanation, new example)
- [ ] New feature (extension, new use case)
- [ ] Documentation (typo, clarification)

## Testing
How did you test these changes?
- [ ] Tested on Windows
- [ ] Tested on Mac
- [ ] Reviewed by another teacher
- [ ] Used in actual workshop

## Context
- Problem being solved: 
- Your teaching experience with this: 
- Classroom size and level: 

## Screenshots (if applicable)
Add before/after screenshots

## Checklist
- [ ] My changes follow the style guide
- [ ] I've tested this thoroughly
- [ ] Documentation is updated
- [ ] No new dependencies added
- [ ] Beginner-friendly approach maintained
```
